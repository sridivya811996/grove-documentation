## 1. Design Principles

### 1.1 Core Philosophy

| Principle | Description |
|-----------|-------------|
| **Standalone Value** | Users can register without a community; the app has value even solo |
| **Frictionless Joining** | One tap to join via invite link; no approval needed by default |
| **Admin First** | First person to create a community becomes admin automatically |
| **Progressive Disclosure** | Show features as users need them, not all at once |
| **WhatsApp Familiar** | Borrow patterns users already know from WhatsApp |

### 1.2 Key Questions Answered

| Question | Decision | Rationale |
|----------|----------|-----------|
| Can users exist without a community? | **Yes** | Reduces friction; they can explore, then join/create |
| Can users be in multiple communities? | **Yes** | A person may be in park group + sports club |
| Who can create communities? | **Any registered user** | No gatekeeping; growth > control |
| Default join mode? | **Invite-only (private)** | Matches WhatsApp behavior; safe default |
| Can communities be public? | **Yes (opt-in)** | For larger orgs wanting discoverability |

### 5.3 Role Descriptions

#### Owner
- **Who**: The person who created the community
- **Special Powers**:
  - Only one who can delete the community
  - Only one who can transfer ownership
  - Cannot be removed or demoted
- **Use Case**: Founder protection

#### Admin
- **Who**: Trusted leaders appointed by owner/admins
- **Special Powers**:
  - Full management except deletion
  - Can manage other admins (but not owner)
  - Can change all settings
- **Use Case**: Co-organizers, committee heads

#### Moderator
- **Who**: Helpers who manage content
- **Special Powers**:
  - Can manage events and messages
  - Can approve join requests
  - Limited settings access
- **Use Case**: Active volunteers, event coordinators

#### Member
- **Who**: Everyone who joins
- **Special Powers**:
  - Basic participation
  - Subject to community settings
- **Use Case**: General community members

## 7. Edge Cases & Business Rules

### 7.1 User Edge Cases

| Scenario | Handling |
|----------|----------|
| User joins community they're already in | Show "You're already a member" |
| User tries to join with expired link | Show error, suggest contacting admin |
| User deleted their account, rejoins | Fresh start, new user record |
| User in 20+ communities | Allow (no limit in MVP) |
| User has no communities and deletes app | Data preserved, can come back |

### 7.2 Community Edge Cases

| Scenario | Handling |
|----------|----------|
| Owner leaves community | Must transfer ownership first |
| Only admin leaves | Community continues, members can't change settings |
| All members leave | Community archived, owner notified |
| Duplicate community names | Allowed (identified by ID) |
| Community has 0 events/messages | Valid state, show empty states |

### 7.3 Invite Link Edge Cases

| Scenario | Handling |
|----------|----------|
| Link shared after limit reached | Show "This link has reached its limit" |
| Link used after expiration | Show "This link has expired" |
| Link from deleted community | Show "This community no longer exists" |
| Same user clicks link twice | No duplicate, show "already a member" |
| User clicks link while logged out | Store link, prompt after login |

### 7.4 Permission Edge Cases

| Scenario | Handling |
|----------|----------|
| Admin demotes themselves | Allowed if other admins exist |
| Owner demotes all admins | Allowed (owner is enough) |
| Last moderator is demoted | Allowed (admins can moderate) |
| Member promoted past moderator | Must go: member â†’ mod â†’ admin |
